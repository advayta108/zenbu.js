# Plugins overview

A **plugin** is a directory with a [`zenbu.plugin.json` manifest](/api/manifest/zenbu-plugin-json). Plugins are the only unit of code organization the framework cares about — your application is itself a plugin, and any third-party plugin uses the exact same shape and APIs.

```
my-plugin/
├── zenbu.plugin.json     # the manifest
├── package.json
├── tsconfig.json
├── setup.ts              # optional — runs once on plugin install
├── vite.config.ts        # optional — for plugins with renderer views
├── src/
│   ├── main/             # services + schema + events + preload
│   │   ├── schema.ts
│   │   ├── events.ts
│   │   ├── preload.ts
│   │   └── services/
│   │       ├── app.ts
│   │       └── tracker.ts
│   ├── renderer/         # UI source rendered as views
│   └── content/          # advice + content scripts
└── db/
    └── migrations/       # generated; commit them
```

## What a plugin contributes

A plugin can contribute any of:

- **Services** — long-lived classes registered with the runtime (most common). Glob in `manifest.services`.
- **A database section** — a typed schema and migrations. `manifest.schema`.
- **Event topics** — typed server-to-client events. `manifest.events`.
- **A preload** — eager boot-time data. `manifest.preload`.
- **Views** — Vite-built renderer apps mounted as iframes. Registered at runtime by services.
- **Content scripts and advice** — code injected into other plugins' views. Registered at runtime.
- **A setup script** — one-time host setup (install deps, download binaries). `manifest.setup`.

A plugin can contribute as little as a single field-less service, or as much as a full sub-application.

## Plugins are just paths in `config.json`

The framework reads `~/.zenbu/config.json` (the path is overridable; see [App config](/boot/app-config)) to find the active plugin set:

```json
{
  "db": "./.zenbu/db",
  "plugins": [
    "/Users/you/projects/my-app/zenbu.plugin.json",
    "/Users/you/.zenbu/plugins/notes-plugin/zenbu.plugin.json",
    // "/Users/you/.zenbu/plugins/disabled-plugin/zenbu.plugin.json"
  ]
}
```

- The order is meaningful: services are imported in this order, but service evaluation order is determined by the dependency graph.
- A line that's commented out (line starts with `//`) is a **disabled** plugin — preserved so toggling on/off doesn't lose ordering.
- Paths are absolute — the framework resolves them via the [zenbu loader chain](/runtime/hot-reload#the-main-process-loader-chain).

## Your application = a plugin

When you scaffold a new app with `create-zenbu-app`, the resulting directory is itself a plugin. Its manifest's path is what gets added to `config.json`. The application's `package.json#main` points at `@zenbujs/core/setup-gate.mjs`, which is what Electron actually launches; the framework reads `config.json`, imports your manifest, and starts your services.

There's no privileged "host" plugin. If you depend on another plugin's service, the dep map looks identical for both directions.

## How plugins compose

Two plugins can:

- **Read each other's database sections.** As long as both sections exist in the registry (i.e. both plugins are installed), `root.plugin["other-plugin"].something` is typed and available. Cross-plugin **writes** should go through RPC, not direct mutation — see [Using on the server](/db/using-on-server#cross-section-writes).
- **Subscribe to each other's events.** Same registry-typed proxy.
- **Depend on each other's services.** As a typed dep (`Service.with({ x: OtherService })`) or as a string-key dep (`Service.with({ x: "other-service" })`).
- **Advise each other's functions.** See [Advice](/advice/overview).
- **Inject content scripts into each other's views.** See [Content scripts](/advice/content-scripts).

The framework intentionally doesn't have a permission model on these — Zenbu is an end-user, single-machine system, not a multi-tenant SaaS. Trust is established at install time (the user pasted a path into config.json) and is total from then on.
