# Installing & managing plugins

For end users, plugin install is "add a path to `~/.zenbu/config.json`." The framework provides UI and CLI helpers to make that less manual.

## The user's `config.json`

Every Zenbu app reads `~/.zenbu/config.json` (or whatever `ZENBU_CONFIG_PATH` points at):

```jsonc
{
  // Where the database lives. Relative paths are relative to config.json's directory.
  "db": "./.zenbu/db",

  // Plugins, in priority order. Lines starting with // are disabled but preserved.
  "plugins": [
    "/Users/you/projects/my-app/zenbu.plugin.json",
    "/Users/you/.zenbu/plugins/notes-plugin/zenbu.plugin.json"
    // "/Users/you/.zenbu/plugins/calendar-plugin/zenbu.plugin.json"
  ]
}
```

The file is JSONC (comments + trailing commas allowed) precisely so you can comment plugins out. The framework's edit tools preserve comments.

## Install via the registry

Many plugins are listed in the framework's plugin registry — a JSON-Lines file at the repo root (`registry.jsonl`):

```jsonl
{"name":"notes-plugin","title":"Notes","description":"Markdown notes","repo":"github.com/example/zenbu-notes-plugin"}
{"name":"calendar-plugin","title":"Calendar","description":"Local calendar","repo":"github.com/example/zenbu-calendar-plugin"}
```

The host shell renders this as a list with one-click install. Behind the scenes, install is:

1. Clone the repo to `~/.zenbu/plugins/<name>/`.
2. Run the plugin's `setup.ts` if declared.
3. Append the manifest path to `config.json`.
4. Trigger a runtime reload (or relaunch if `setup.ts` modified `node_modules`).

## Install manually

```bash
git clone https://github.com/example/zenbu-notes-plugin ~/.zenbu/plugins/notes-plugin
nr zen install ~/.zenbu/plugins/notes-plugin     # runs setup, adds to config.json
```

## Toggle on/off

Enable / disable without removing files (preserves order, preserves the line):

```bash
nr zen plugin toggle notes-plugin off
nr zen plugin toggle notes-plugin on
```

The host shell exposes this as a switch in the plugin manager UI.

## Uninstall

```bash
nr zen plugin remove notes-plugin       # removes from config.json
rm -rf ~/.zenbu/plugins/notes-plugin    # delete files
```

The framework intentionally separates these — removing from `config.json` is the safe default; deleting files is destructive and you should confirm before doing it.

## Inspecting the plugin set

```bash
nr zen plugin list                      # all configured plugins, with enabled state
nr zen plugin info notes-plugin         # manifest fields, registered services
```

From a service / RPC consumer, the [`InstallerService`](/api/core/runtime#built-in-services) exposes:

- `getInstalledPlugins()` — all plugins from `config.json`, with their resolved service files.
- `listPluginsWithStatus()` — same list, including the disabled ones.
- `addPluginToConfig(manifestPath)`, `removePluginFromConfig(manifestPath)`, `togglePlugin(manifestPath, enabled)`.
- `runSetup(onProgress)` — re-run a plugin's setup script.
- `getPluginRegistry()` — read `registry.jsonl`.

## Updates

Plugin updates are git pulls:

```bash
cd ~/.zenbu/plugins/notes-plugin
git pull
nr zen plugin reload notes-plugin
```

The framework's [live-source distribution](/build/live-source-adapter) extends this to your application code itself: app updates flow over git the same way.

## Permissions (planned)

Currently, every plugin has full filesystem and network access — same as any Node module. A near-term enhancement is a manifest field declaring required capabilities:

```json
{
  "setup": {
    "script": "./setup.ts",
    "version": 1,
    "permissions": ["fs:write", "net:fetch"]
  }
}
```

The host shell would prompt before running setup. This is on the [open questions](/reference/open-questions) page; not yet implemented.

## What if two plugins conflict?

Plugins **cannot** share a `name` or a `static key` for any service. The runtime detects this at boot:

- Same plugin name in two manifests → second is rejected with an error.
- Same service key across two plugins → the second registration replaces the first (treated as HMR).

In practice, conflicts in the wild are rare, because plugin names tend to be unique organically.
