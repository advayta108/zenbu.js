# `config.json`

Every Zenbu app reads a single `config.json` file at boot to know what plugins to load and where the database lives.

## Resolution

The framework resolves the config path in this order:

1. The `ZENBU_CONFIG_PATH` env var, if set.
2. `<project-root>/config.json`.
3. `~/.zenbu/config.jsonc`.
4. `~/.zenbu/config.json`.

In dev (`nr dev`), the framework `chdir`s into your project root and uses option 2. In production live-source mode, the launcher sets `ZENBU_CONFIG_PATH` to the apps-dir copy.

## Schema

```jsonc
{
  // Where the database lives, relative to config.json's directory or absolute.
  // Required.
  "db": "./.zenbu/db",

  // Plugins, in priority order. Each entry is an absolute path to a
  // zenbu.plugin.json. Lines starting with // are disabled but preserved.
  // Required (can be empty array).
  "plugins": [
    "/Users/you/projects/my-app/zenbu.plugin.json"
  ]
}
```

Both fields are required. The framework refuses to boot if either is missing.

### `db`

Where the SQLite-style document database lives on disk. Resolution rules:

- Absolute path → used as-is.
- Relative path → resolved against the directory containing `config.json`.

The directory is created if it doesn't exist; existing data is preserved across reboots. Multiple Zenbu apps **must not share the same `db`** — there's a cross-process lock to enforce this.

### `plugins`

Ordered list of absolute paths to `zenbu.plugin.json` manifests.

- **Order matters for** import order (services in earlier plugins are imported first), but **not for** evaluation order — that's the dep graph's job.
- **Comments** (lines starting with `//`) are valid and preserved across edits. The host shell's plugin manager toggles enable/disable by adding/removing the `//` prefix.
- Paths must be absolute. Relative paths are rejected at boot with an error.

## JSONC support

`config.json` is parsed as JSONC: comments and trailing commas are tolerated. The framework's plugin manager UI relies on this so toggling a plugin off doesn't lose its position in the list.

## Multiple apps on one machine

If you're running multiple Zenbu apps on one machine, each app should:

- Have its own `db` location (typically inside the app's project root).
- Have its own `config.json`.
- Either be in different project directories (and use option 2), or set `ZENBU_CONFIG_PATH` per app.

The shared `~/.zenbu/config.json` is for users who want **one** persistent installed Zenbu environment ("my Zenbu") across boots — e.g., the development host that loads the framework's own dev plugins.

## Editing programmatically

The [`InstallerService`](/plugins/installing-and-managing#inspecting-the-plugin-set) exposes:

- `addPluginToConfig(manifestPath)`
- `removePluginFromConfig(manifestPath)`
- `togglePlugin(manifestPath, enabled)`

These rewrite `config.json` while preserving comments and ordering.

## CLI access

```bash
nr zen config get db                  # print the db path
nr zen config set db "./other-path"   # change the db path
nr zen plugin list                    # list plugins
```

## Defaults

The scaffold (`create-zenbu-app`) emits a `config.json` with one plugin (your own app) and `db: ./.zenbu/db`. Most apps never need to edit this manually.
