# Hot reload

Zenbu's HMR is unusual in that it spans **the main process, the renderer process, plugin code, and content scripts**. There's a separate mechanism for each, but as a service author you only ever need to understand the main-process one.

## What hot-reloads, and how

| What | Mechanism | Reloads on save? |
| --- | --- | --- |
| Main-process service code | [dynohot](https://github.com/laverdet/dynohot) (proxied imports) | yes — full lifecycle re-run |
| Database schema | DbService re-evaluates with new sections | yes — auto-applied as migration delta |
| RPC router | RpcService re-derives `runtime.buildRouter()` | yes — renderer reconnects |
| Renderer code | Vite (per view) | yes — React Refresh + module HMR |
| Content scripts | Vite (re-runs the prelude) | yes — view reloads |
| Advice registrations | [`registerAdvice`](/advice/function-advice) re-runs | yes — affected views reload |
| `setup-gate.ts` framework code | not reloaded | no — restart the dev server |
| `loaders/zenbu.ts` (the Node loader) | not reloaded | no — restart the dev server |

For 99% of plugin code, the answer is "yes, just save and watch."

## The main-process loader chain

Every import inside the main process passes through four loaders, in order:

1. **Zenbu loader** — resolves `zenbu:plugins?config=...` and `zenbu:barrel?manifest=...` virtual modules into a list of side-effect imports (`import "...service-file..."`).
2. **TSX** — compiles TypeScript to JavaScript.
3. **Advice transform** — Babel pass that rewrites top-level functions so [`advise()`](/advice/function-advice) can wrap them.
4. **Dynohot** — wraps every module in a hot-reload proxy that watches the file on disk.

When you save `src/main/services/my-service.ts`:

1. Dynohot's file watcher fires.
2. The module's import graph is invalidated upward to every file that re-exports it.
3. The next reference to a top-level binding (or an explicit `import.meta.hot?.accept()`) triggers re-evaluation.
4. Re-evaluation hits `runtime.register(MyService, import.meta)` again, which the runtime treats as a slot replacement and triggers the [lifecycle](/runtime/lifecycle).

## Sentinel: `import.meta` on `runtime.register`

The second argument to `runtime.register(MyService, import.meta)` does two things:

- It tells the runtime which module the service comes from, so a re-evaluation of *that exact module* is what triggers a reload (vs e.g. a re-import from a barrel file).
- It calls `importMeta.hot?.accept()` internally, which tells dynohot "I handle my own state across reloads — don't propagate up the dependent chain just to reload me."

If you forget `import.meta`, hot reload still mostly works (the runtime falls back to detecting same-key registrations), but you'll see duplicate registrations in the trace and the dependent subgraph will reload more aggressively than necessary.

## When reload doesn't take

If you save a file and nothing happens, work through this list:

1. **Is the file actually imported by a service?** The Zenbu loader only generates barrel imports for files that match the `services:` glob in your manifest. A new file in `src/main/utils/` won't trigger a reload until something imports it.
2. **Is the file inside a `node_modules` or `dist` directory?** Those are deliberately excluded from dynohot to prevent leaks.
3. **Is `runtime.register(...)` actually called from this file?** If your service file just defines a class but doesn't register it, saving it changes nothing observable.
4. **Did the loader chain crash?** Check the dev console for "module evaluation failed" — usually a syntax error or a missing import.
5. **Are you saving fast enough that two saves overlap?** The runtime queues reloads, but if `start()` takes longer than the inter-save interval you can see the *intermediate* state stick around. Slow down or shorten `start()`.

## Renderer HMR

Each [view](/views/overview) is its own Vite dev server with React Refresh enabled. The framework injects a few plugins into every view's Vite config (the advice prelude, content-script discovery, theme stylesheet) — you don't add them yourself. Standard Vite HMR rules apply.

A few framework-specific behaviors:

- **Database schema changes propagate to the renderer.** When DbService re-evaluates with a new schema, every view's replica reconnects and fetches the new root.
- **RPC API changes propagate via reconnect.** When RpcService re-derives the router, all views' websockets are dropped and re-established. `useRpc()` keeps working — you just see a brief reconnecting state.
- **Advice registrations are propagated as a `advice:reload` event.** Any view in the affected scope reloads itself (full page reload, not React refresh — advice is too low-level).

## Reload from RPC

If you want a programmatic reload (e.g. after a plugin's `setup.ts` ran), the [`runtime` service](/runtime/runtime-control) exposes:

```typescript
await rpc.runtime.reload()           // reload all services in place
await rpc.runtime.quitAndRelaunch()  // hard restart
```

Use `reload` for "I changed code/config and want all services to re-evaluate without dropping the window." Use `quitAndRelaunch` for "node_modules changed; the loader cache is stale; we need a clean process."

## What it costs

For our reference plugin set:

- A leaf service reload (one service, no dependents): **2-5ms**.
- A `DbService` reload (everything depends on it): **80-200ms** depending on schema diff size.
- A renderer view reload triggered by an advice change: **300-500ms** (full page).

If your reload takes longer than the leaf-service number, the bottleneck is almost always something the service is doing in `start()` that should be moved to a method. Trace it with `nr zen profile` or via the boot bus events that the [`runtime` service](/runtime/runtime-control) exposes.
