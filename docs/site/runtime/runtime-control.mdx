# Runtime control

The framework ships a built-in service called `RuntimeControlService` (key: `"runtime"`) that exposes a small RPC surface for tooling. You almost never need to use it from your own services — the typical caller is an "admin / dev" view in your app.

## RPC

```typescript
const rpc = useRpc()

// Tear down and re-evaluate every service. The websocket reconnects mid-call.
await rpc.runtime.reload()

// Quit and relaunch the Electron app. Use after a setup.ts modified node_modules.
await rpc.runtime.quitAndRelaunch()
```

Both return `{ ok: true }` and are fire-and-forget — the wire dies before the response, by design.

## Introspecting state

For UIs that visualize the runtime (e.g. a "service health" panel), depend on the runtime singleton directly:

```typescript
import { runtime } from "@zenbujs/core/runtime"

export class HealthService extends Service {
  static key = "health"

  async list(): Promise<Array<{ key: string; status: string; error?: string }>> {
    return runtime.getAllKeys().map(key => {
      const slot = runtime.getSlot(key)!
      return {
        key,
        status: slot.status,                           // "blocked" | "evaluating" | "ready" | "failed"
        error: slot.error ? String(slot.error) : undefined,
      }
    })
  }
}
```

`getSlot()` returns `undefined` for keys that aren't registered. The `status` enum is stable; new states will only ever be additions.

## Lifecycle hooks

```typescript
import { runtime } from "@zenbujs/core/runtime"

const off = runtime.onReconciled(changedKeys => {
  console.log("services that just settled:", changedKeys)
})

// Later, to stop listening:
off()
```

`onReconciled` fires every time the runtime finishes a reconcile pass — boot, hot-reload, optional-dep flips, etc. It's a good hook for telemetry or "rebuild caches when the graph stabilizes" logic.

## What you should not do

- **Don't call `runtime.shutdown()` from a service.** Use `app.quit()` (from Electron) or `rpc.runtime.quitAndRelaunch()`. `shutdown()` is the runtime's response to those, not the trigger.
- **Don't reach into other services via `runtime.get(...)` instead of declaring a dep.** See [Dependencies — anti-pattern: dynamic deps](/runtime/dependencies#anti-pattern%3A-dynamic-deps).
- **Don't iterate `runtime.getAllKeys()` to derive behavior.** That couples your service to the entire plugin set. Treat it as observability only.

For the full surface of `runtime`, see the [API reference](/api/core/runtime#runtime-1).
