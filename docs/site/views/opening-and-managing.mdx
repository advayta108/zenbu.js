# Opening & managing views

The `Window` service (key: `"window"`) is the public API for mounting and arranging views.

```typescript
import { Service, runtime } from "@zenbujs/core/runtime"
import { Window } from "@zenbujs/core/services"

export class CommandsService extends Service.with({ window: Window }) {
  static key = "commands"
}
```

## `openView({ scope, windowId?, query? })`

Mount a registered view in a window. Returns `{ windowId }`.

```typescript
await this.ctx.window.openView({ scope: "notes" })
```

| Argument | Meaning |
| --- | --- |
| `scope` | The view's `id` from registration. Required. |
| `windowId` | Which window to mount in. Defaults to `MAIN_WINDOW_ID`. |
| `query` | Optional record. Appended to the iframe URL as a query string. |

`openView` is idempotent at the (windowId, scope) level: opening the same scope in the same window twice is a no-op.

If you mount a different scope into a window that already had one, the old view is unmounted (its WebContents are closed) and the new view is mounted in its place. Use a different `windowId` to keep both.

### Custom queries

```typescript
await this.ctx.window.openView({
  scope: "notes",
  query: { initialNote: "abc123" },
})
```

The query string lands in the iframe URL: `…/index.html?initialNote=abc123&wsPort=…`. The view reads it via `new URL(window.location.href).searchParams`.

This is the canonical way to pass per-mount initial state to a view. Persistent state goes in the database.

## `focusWindow(windowId)`

Bring a window to the front:

```typescript
await this.ctx.window.focusWindow(windowId)
```

If the window is hidden, it's shown first.

## `pickFiles()` / `pickDirectory()`

System file pickers, scoped to the focused window:

```typescript
const files = await this.ctx.window.pickFiles()      // string[] | null
const dir = await this.ctx.window.pickDirectory()    // string | null
```

Both return `null` if the user cancels.

## `openExternal(url)` / `openPath(path)`

Forward a URL to the user's default browser, or a path to the OS file handler:

```typescript
await this.ctx.window.openExternal("https://example.com")
await this.ctx.window.openPath("/Users/me/Downloads")
```

## `copyToClipboard(text)`

Synchronous; writes text to the system clipboard:

```typescript
this.ctx.window.copyToClipboard("hello")
```

## Multi-window apps

The framework supports multiple top-level windows. Each is identified by a `windowId` (a string of your choice). Common patterns:

```typescript
// Main window — implicit
await this.ctx.window.openView({ scope: "notes" })

// Companion window for a specific item
await this.ctx.window.openView({
  scope: "note-detail",
  windowId: `note-${id}`,
  query: { id },
})

// Settings in its own window
await this.ctx.window.openView({
  scope: "settings",
  windowId: "settings",
})
```

The framework creates the window lazily on first `openView`. Window bounds (position + size) are persisted by the `core` section; reopening a window restores its last position.

## RPC visibility

`Window` is a regular service; its public methods are auto-exposed as RPC. So a renderer can directly:

```tsx
import { useRpc } from "@zenbujs/core/react"

function Settings() {
  const rpc = useRpc()
  return (
    <button onClick={() => rpc.window.openView({ scope: "notes" })}>
      Notes
    </button>
  )
}
```

For most apps, this is the right level of access. If you want to gate it behind a higher-level command, write a small plugin service that wraps `window` and exposes only what you want callable.
