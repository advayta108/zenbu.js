# Views overview

A **view** is a Vite-built React app, mounted in a Zenbu window as a `WebContentsView` (Electron's iframe replacement). Views are how plugins contribute UI.

A few invariants that drive the rest of the design:

- **Each view is its own renderer process.** Crash isolation, security isolation, and the ability to ship plugins that share zero JavaScript with each other.
- **Each view has its own Vite dev server.** Per-view HMR, per-view Vite plugin set.
- **Views share the same websocket-based RPC + DB + events.** A view can read your service's database section and call your service's RPC even though it's running in a different process.
- **Plugins can install UI into other plugins' views.** Via [content scripts](/advice/content-scripts) and [advice](/advice/function-advice).

## What a view looks like on disk

A view is a normal React + Vite app:

```
src/renderer/
├── index.html
├── main.tsx
├── App.tsx
└── app.css
```

`main.tsx` is the entrypoint. `index.html` is the shell. `vite.config.ts` (one level up, shared across views in the plugin) declares the build.

The framework injects a few mandatory plugins into every view's Vite config — the [advice prelude](/advice/overview), content-script discovery, framework resolve hooks. You don't add them yourself.

## Registering a view

A plugin registers a view from a service. The recommended pattern uses the built-in `Views` service:

```typescript
import path from "node:path"
import { fileURLToPath } from "node:url"
import { Service, runtime } from "@zenbujs/core/runtime"
import { Views } from "@zenbujs/core/services"

export class NotesViewService extends Service.with({ views: Views }) {
  static key = "notes:view"

  async start() {
    const here = path.dirname(fileURLToPath(import.meta.url))
    this.useDisposable("register-view", () =>
      this.ctx.views.register({
        id: "notes",
        root: path.resolve(here, "..", "..", "renderer"),
        viteConfig: path.resolve(here, "..", "..", "..", "vite.config.ts"),
        panel: { sidebar: true, label: "Notes" },
      }),
    )
  }
}

runtime.register(NotesViewService, import.meta)
```

Behind the scenes:

1. The framework boots a Vite dev server rooted at `root`.
2. It allocates an ephemeral local port and starts the server.
3. It records the URL in the view registry (a database section under `core.lastKnownViewRegistry`).
4. The host shell's sidebar reads the registry and renders the view as an option.

When the user opens this view, Electron creates a `WebContentsView` and points it at the Vite URL with a query string carrying `?wsPort=&wsToken=&windowId=&scope=notes`. The view's `<ZenbuProvider>` reads those query params to wire up the websocket.

## Default view from the manifest

If your plugin has only one view and you want zero-config registration, set `manifest.uiEntrypoint`:

```json
{
  "name": "notes-plugin",
  "uiEntrypoint": "src/renderer"
}
```

The framework auto-registers a view at `<plugin-root>/src/renderer/` with the scope name equal to the plugin's `name`. No service required.

## Opening a view

```typescript
import { Service, runtime } from "@zenbujs/core/runtime"
import { Window } from "@zenbujs/core/services"

export class CommandService extends Service.with({ window: Window }) {
  static key = "commands"

  async openNotes() {
    await this.ctx.window.openView({ scope: "notes" })
  }
}
```

`window.openView({ scope, windowId?, query? })` mounts the registered view in a window. If `windowId` isn't provided, it uses the main window. `query` (key/value record) is appended to the iframe URL — useful for passing initial state.

## Iframe semantics

A view runs in a Chromium iframe. Practical implications:

- `localStorage` and `sessionStorage` are scoped to the view's origin (the framework gives each view a unique ephemeral port, so each view gets its own).
- Cross-origin resource sharing applies. The framework's local-file protocol (`local://`) handles your filesystem access; remote fetches go through normal browser CORS.
- The DOM is sandboxed. A view can't reach into another view's DOM.
- Right-click → Inspect Element works out of the box. So does Cmd+Alt+I (Mac) / Ctrl+Shift+I (Linux/Windows) for devtools.
