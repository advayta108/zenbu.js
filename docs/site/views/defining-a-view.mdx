# Defining a view

Views are registered imperatively from a service. The signature:

```typescript
this.ctx.views.register({
  id: string,                  // scope — must be globally unique across plugins
  root: string,                // absolute path to the renderer directory
  viteConfig?: string | false, // absolute path to vite.config.ts, or false to skip
  panel?: PanelOptions,        // host-shell display metadata
})
```

`register` returns an unregister function — pair it with `useDisposable` so it tears down on reload.

## The minimum

```typescript
this.useDisposable("view", () =>
  this.ctx.views.register({
    id: "notes",
    root: "/abs/path/to/src/renderer",
  }),
)
```

This:

- Boots a Vite dev server rooted at `root`.
- Picks an ephemeral port.
- Registers the URL in the view registry.

## With a `vite.config.ts`

Most plugins want their own Vite config (Tailwind, custom plugins, etc.):

```typescript
this.ctx.views.register({
  id: "notes",
  root: "/abs/path/to/src/renderer",
  viteConfig: "/abs/path/to/vite.config.ts",
})
```

The framework resolves your config and merges in the mandatory framework plugins (advice prelude, content-script discovery, theme stylesheet) so you don't have to add them.

If you pass `viteConfig: false`, **only** the framework plugins are used. This is rare; useful for trivial views with no extra dependencies.

## Recommended `vite.config.ts`

For a typical view, you can use the helper:

```typescript
// vite.config.ts
import { defineZenbuViewConfig } from "@zenbujs/core/vite"

export default defineZenbuViewConfig()
```

`defineZenbuViewConfig()` resolves `@vitejs/plugin-react` and the framework's CSS preset against the framework's own `node_modules`, so plugins don't have to install React-Vite tooling. If you want to extend it:

```typescript
import { defineZenbuViewConfig } from "@zenbujs/core/vite"
import tailwind from "@tailwindcss/vite"

export default defineZenbuViewConfig({
  plugins: [tailwind()],
  resolve: {
    alias: { "@": "/src/renderer" },
  },
})
```

Pass any standard Vite config; it's deep-merged onto the framework defaults.

## Resolving paths

`root` and `viteConfig` are absolute. The most reliable way to compute them from a service file:

```typescript
import path from "node:path"
import { fileURLToPath } from "node:url"

const here = path.dirname(fileURLToPath(import.meta.url))
const root = path.resolve(here, "..", "..", "renderer")
const viteConfig = path.resolve(here, "..", "..", "..", "vite.config.ts")
```

The plugin manifest's directory is *not* automatically passed to your service — but you can find it by walking up from `import.meta.url` looking for `zenbu.plugin.json`:

```typescript
import { findPluginRoot } from "@zenbujs/core/runtime"

const root = path.resolve(findPluginRoot(import.meta), "src/renderer")
```

`findPluginRoot(import.meta)` is a small helper exported alongside the runtime. It walks up from the caller's URL until it finds a `zenbu.plugin.json`.

## Aliases

If multiple views in your plugin should appear under different scopes but share one Vite server, register the primary view with `register(...)` and the aliases with `registerAlias(...)`:

```typescript
this.ctx.views.register({
  id: "notes",
  root: "/abs/path/to/src/renderer",
})
this.ctx.views.registerAlias({
  id: "notes:settings",
  reloaderId: "notes",
  pathPrefix: "/settings",
})
```

`registerAlias` reuses the existing Vite server but registers a different scope at a sub-path. Useful when multiple "panels" share code.

## Hot reload & re-registration

If your service reloads (you saved its source), the previous registration's teardown runs (returned by `register(...)`), then the new registration runs — same server gets reused if `id` and `root` are unchanged.

Save your renderer source — Vite handles HMR for that, no service work involved.

Save your `vite.config.ts` — the framework restarts the Vite server automatically.

## What if I want to mount a non-React view?

The framework doesn't actually care that the view is React. As long as your `index.html` boots whatever framework you want, it works. The only constraint is you should use `<ZenbuProvider>` (from `@zenbujs/core/react`, but it works fine if you wrap a non-React tree, since it just sets up the websocket connection).

We may add framework-agnostic helpers (`zenbu-vue`, `zenbu-svelte`) once a real demand surfaces.
