# Glossary

A short list. We try to keep names few and consistent.

## Adapter

A bundle-distribution strategy plugged into `zenbu.build.ts` via `adapter:`. The two built-in adapters are `liveSource()` (the default) and `frozenSource()`. See [Build & distribution overview](/build/overview).

## Advice

A function-level wrapper registered via `advise(...)` that wraps, replaces, or hooks an exported function in another plugin's view source. See [Advice overview](/advice/overview).

## Apps directory

The per-user directory where live-source apps store their working copy. Defaults to `~/.zenbu/apps/<name>/`. Overridable via `ZENBU_APPS_DIR`.

## Boot bus

Internal event bus used by the framework to report boot progress to the host shell. Not part of the public API.

## Collection

A paginated, lazily-loaded array stored as JSONL files under the database directory. Use `f.collection(itemSchema)` in your schema. See [Collections](/db/collections).

## Content script

A TS/TSX module injected into another plugin's view via `inject(...)`. Runs alongside the view's own code. See [Content scripts](/advice/content-scripts).

## Dep / dependency

A service another service requires. Declared via `Service.with({ name: OtherService })`. See [Dependencies](/runtime/dependencies).

## DbService

The framework's built-in database service. Owns the on-disk database, broadcasts updates to renderer replicas. Key: `"db"`.

## Disposable

A resource registered via `useDisposable("name", fn)` on a service. The framework runs the teardown automatically on reload and shutdown. See [Lifecycle](/runtime/lifecycle).

## Event

A typed server-to-client signal declared in a plugin's `events.ts` and emitted via the `Events` service. See [Events](/rpc/events).

## Frozen-source

The traditional Electron build mode — pre-compiled JS, asar-bundled, no toolchain. Activated by `adapter: frozenSource()`. See [Frozen-source adapter](/build/frozen-source-adapter).

## HMR (hot module reload)

Source-code changes propagating to a running app without a restart. In Zenbu, services reload via dynohot, schemas via DbService, renderers via Vite. See [Hot reload](/runtime/hot-reload).

## Kyju

The internal name of the database engine. The public surface lives at `@zenbujs/core/db`; the engine is at `@zenbu/kyju`. Plugin code never imports the latter directly.

## Live-source

The default distribution mode — ships TypeScript source, bundles `pnpm` + `bun`, updates over git. Activated by `adapter: liveSource(...)`. See [Live-source adapter](/build/live-source-adapter).

## Loader chain

The four Node module loaders the framework registers on boot: zenbu loader (virtual modules), tsx (TS compile), advice transform, dynohot (HMR). See [Hot reload](/runtime/hot-reload#the-main-process-loader-chain).

## Manifest

The plugin's `zenbu.plugin.json` file. See [Manifest schema](/plugins/manifest).

## Migration

A declarative diff applied to the on-disk database to bring it in sync with the current schema. Generated by `nr zen db generate`. See [Migrations](/db/migrations).

## Mirror repo

A separate git repo that holds your app's published source. End users `git pull` from this repo to receive updates. Used in live-source mode. See [Live-source adapter](/build/live-source-adapter).

## Namespace

The name a plugin's contributions show up under: `root.plugin["<name>"]`, `events["<name>"]`, `preloads["<name>"]`. Derived from `manifest.name`.

## Optional dep

A dep wrapped in `optional(...)` that's allowed to be missing. Resolves to `Instance | undefined` in `this.ctx`. See [Optional dependencies](/runtime/dependencies#optional-dependencies).

## Plugin

A directory with a `zenbu.plugin.json`. The unit of distribution and composition. See [Plugins overview](/plugins/overview).

## Preload

A boot-time-resolved value declared in a plugin's `preload.ts`, available to all services and renderers. See [Preloads overview](/preloads/overview).

## Registry (TypeScript)

The augmentation point at `@zenbujs/core/registry` that gives `useDb`, `useRpc`, `useEvents`, `usePreload` their types. Populated by `zen link`. See [Typed registry](/rpc/typed-registry).

## Replica

The in-memory copy of the database held by each renderer. Updates optimistically; reconciles with main-process broadcasts. See [Database overview](/db/overview).

## Reloader

The Vite dev-server pool. Each registered view gets its own Vite server. Internal; users interact via `Views.register(...)`.

## RPC

Remote procedure call. Any `async` method on a registered service whose name doesn't start with `_` is automatically RPC-callable. See [RPC overview](/rpc/overview).

## Runtime

The singleton that owns every service in a process. Imported as `runtime` from `@zenbujs/core/runtime`. See [Runtime overview](/runtime/overview).

## Scope

A view's id, also used as the targeting key for advice and content scripts. Don't confuse with namespace (which is the plugin's name).

## Section

A plugin's portion of the database, located at `root.plugin["<name>"]`. Each plugin owns one section.

## Service

A long-lived dependency-injected class registered with the runtime. The main programming primitive on the main process. See [Services](/runtime/defining-a-service).

## setup-gate

The framework's main-process entrypoint. `package.json#main` points at `@zenbujs/core/dist/setup-gate.mjs` in live-source apps. See [setup-gate](/boot/setup-gate).

## View

A renderer-process micro-frontend rendered as a `WebContentsView`. See [Views overview](/views/overview).

## ZenbuRegister

The module-augmentation interface at `@zenbujs/core/registry`. See [Typed registry](/rpc/typed-registry).
