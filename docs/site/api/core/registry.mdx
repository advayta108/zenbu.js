# `@zenbujs/core/registry`

```typescript
import type { ZenbuRegister } from "@zenbujs/core/registry"
```

The registry is **type-only** — there's nothing to import at runtime. Its single export is an `interface` that plugin types are merged into via TypeScript's [module augmentation](https://www.typescriptlang.org/docs/handbook/declaration-merging.html#module-augmentation).

## `ZenbuRegister`

```typescript
export interface ZenbuRegister {}
```

The augmentation point. By default, empty. After [`zen link`](/api/cli/zen#zen-link) generates `<your-app>/types/zenbu-register.ts`, it's filled in:

```typescript
// Generated. Do not edit manually.
declare module "@zenbujs/core/registry" {
  interface ZenbuRegister {
    db: DbRoot                          // shape of `useDb()`
    rpc: ServiceRouter                  // shape of `useRpc()`
    events: PluginEvents                // shape of `useEvents()`
    preloads: Preloads                  // shape of `usePreload()`
  }
}
```

Each field's value is a generated type that imports every registered plugin's source and merges them.

## Falling back to core types

If `ZenbuRegister` has no augmentation, the framework's hooks fall back to:

| Hook | Fallback |
| --- | --- |
| `useDb()` | `{ plugin: { core: CoreSchemaRoot } }` |
| `useRpc()` | `CoreServiceRouter` (empty — built-in services don't expose RPC by default) |
| `useEvents()` | `CoreEvents` |
| `usePreload()` | `CorePreloads` (empty) |

These fallbacks are **internal types**, not part of the public contract. They exist so the package compiles and runs with no plugins registered, not because plugin code should rely on them. If you want fully-typed hooks, run `zen link`.

## What's *not* public

- The fallback types (`CoreSchemaRoot`, `CoreEvents`, `CorePreloads`, `CoreServiceRouter`) — internal.
- `ResolvedDbRoot`, `ResolvedServiceRouter`, `ResolvedEvents` — conditional types used internally to resolve `ZenbuRegister`. Use `ZenbuRegister extends { db: infer T } ? T : never` in your own code instead.

## Adding to the registry from a plugin

You **don't**. The registry is per-app, generated by `zen link` from the app's `config.json`. A plugin doesn't directly augment the registry — its augmentation happens via the generator picking up its `schema.ts`, `events.ts`, `preload.ts`, and service files.

If you need to add a custom typed surface to the registry from your plugin, the framework's roadmap includes a `manifest.types` extension point. Until that lands, generate the types in your plugin's own `types/` directory and have the consuming app's `tsconfig.json` reference them.
