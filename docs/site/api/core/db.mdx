# `@zenbujs/core/db`

This is the database's public-API contract. The underlying engine is an internal package called [`@zenbu/kyju`](https://github.com/zenbu-org/zenbu/tree/main/packages/kyju), but **plugin authors only ever import from `@zenbujs/core/db`**. The kyju package may add, remove, or reshape its own exports between minor versions; this re-export wall is what you can rely on.

## Imports

```typescript
import {
  defineSchema,
  f,
  z,
  type InferSchema,
  type Schema,
  type CollectionRef,
  type BlobRef,
} from "@zenbujs/core/db"
```

---

## `defineSchema(shape)`

Declare a section's schema.

```typescript
function defineSchema<TShape extends SchemaShape>(shape: TShape): Schema<TShape>
```

The returned schema is exported from your plugin's schema file and consumed by the framework via the `schema` field in `zenbu.plugin.json`.

### Example

```typescript
import { defineSchema, f, z } from "@zenbujs/core/db"

export const schema = defineSchema({
  count: f.number().default(0),
  user: f.object({ name: z.string() }),
  notes: f.collection(z.object({ id: z.string(), text: z.string() })),
  thumbnail: f.blob({ debugName: "thumbnail" }),
})
```

## `f`

The field constructor. A typed proxy over [zod](https://zod.dev), augmented with `.default(value)` on every constructor and the `f.collection` / `f.blob` extensions.

```typescript
const f: {
  // every zod constructor, with .default(value) and .nullable() chainable
  string():    FieldWithDefault<ZodString>
  number():    FieldWithDefault<ZodNumber>
  boolean():   FieldWithDefault<ZodBoolean>
  array<T>(item: T):     FieldWithDefault<ZodArray<T>>
  object<S>(shape: S):   FieldWithDefault<ZodObject<S>>
  record<K, V>(k: K, v: V): FieldWithDefault<ZodRecord<K, V>>
  // ...etc

  // kyju extensions:
  collection<T>(itemSchema: T, opts?: { debugName?: string }): ZodType<CollectionRef<T>>
  blob(opts?: { debugName?: string }): BlobRef
}
```

### Defaults

```typescript
f.number().default(0)              // number, default 0
f.string().nullable().default(null) // string | null, default null
f.array(z.string()).default([])    // string[], default []
```

## `z`

Re-export of [zod](https://zod.dev). Exposed so plugins don't import `zod` themselves — using `z` from `@zenbujs/core/db` guarantees a single zod instance across all plugins (avoids type-identity drift across plugins that bundle different zod versions).

Use `z` inline as needed:

```typescript
defineSchema({
  status: z.enum(["draft", "published"]),
  meta: z.object({ tags: z.array(z.string()) }),
})
```

## `InferSchema<S>`

Pulls the runtime type out of a schema:

```typescript
import type { InferSchema } from "@zenbujs/core/db"
import { schema } from "./schema"

type Root = InferSchema<typeof schema>
//   ^? { count: number; user: { name: string }; notes: CollectionRef<{ id: string; text: string }>; ... }
```

## `DbService` (server-side)

In any service that depends on `DbService` you have a typed client at `this.ctx.db`. The full surface:

```typescript
class DbService {
  /** Synchronous snapshot of the entire root. */
  read(): Root

  /** Run a transaction against the root. Mutations on the draft are committed; throws roll back. */
  update(fn: (draft: Root) => void | Promise<void>): Promise<void>

  /** Open a collection reader. */
  collection<T>(ref: CollectionRef<T>): {
    all(): Promise<T[]>
    page(offset: number, limit: number): Promise<T[]>
    get(id: string): Promise<T | null>
    delete(id: string): Promise<void>
    replace(id: string, item: T): Promise<void>
  }

  /** Blob storage. */
  blobs: {
    create(data: Uint8Array, opts?: { hot?: boolean }): Promise<string>   // returns blobId
    read(blobId: string): Promise<Uint8Array | null>
    delete(blobId: string): Promise<void>
  }

  /** Force-flush the lagged-persistence queue. Idempotent. */
  flush(): Promise<void>
}
```

### Notes

- `Root` is the discriminated union of every registered plugin's section, generated by [`zen link`](/api/cli/zen). At type-check time you get autocomplete for `root.plugin.<plugin-name>`.
- `update` is **not reentrant**. Calling it inside another `update` deadlocks.
- The `sectionsHash` baked into each replica is an internal detail; renderers reconnect automatically when sections change.

## React hooks (renderer-side)

Imported from `@zenbujs/core/react`:

```typescript
import { useDb, useCollection, useBlob } from "@zenbujs/core/react"
```

### `useDb()` / `useDb(selector, isEqual?)`

```typescript
function useDb(): Root
function useDb<T>(
  selector: (root: Root) => T,
  isEqual?: (a: T, b: T) => boolean,
): T
```

Default equality is shallow. Pass a custom one if your selector returns a complex object you want compared structurally.

### `useCollection(ref)`

```typescript
function useCollection<T>(ref: CollectionRef<T> | null | undefined): {
  items: T[]
  collection: CollectionNode<T> | null
  concat(items: T[]): void
}
```

### `useBlob(blobId)`

```typescript
function useBlob(blobId: string | null | undefined): Uint8Array | null
```

## Removed / not public

- `connectDb` / `connectReplica` — used internally by `<ZenbuProvider>`. Plugin code should never call directly.
- `dbStringify` / `dbParse` — internal transport.
- `effectClient` / `EffectFieldNode` / `EffectClientProxy` — Effect-flavored API. **Not part of the public surface.** A separate `@zenbujs/core/db/effect` subpath may be added for users who want it; until then, the [migration recipe](/recipes/migrate-from-effect) shows the bridge.
- `KyjuError` / `InvalidSessionError` / `VersionMismatchError` / `NotFoundError` / `ReferenceExistsError` — Effect-tagged error types. Public errors are plain `Error` instances with a `cause` chain.
- `applyOperations` / `MigrationOp` — internals of the migration runner. The [`KyjuMigration`](#migrations) shape is public.
- `discoverSections` / `resolveManifestModulePath` — internals of section discovery.
- `connectReplica` — used internally; the renderer's `<ZenbuProvider>` calls it.

## Migrations

The migration types are public:

```typescript
import type { KyjuMigration, MigrationOp } from "@zenbujs/core/db"

const migration: KyjuMigration = {
  name: "add_user_email",
  ops: [
    { op: "add", path: ["user", "email"], value: undefined },
  ],
}
```

`MigrationOp` is the discriminated union:

```typescript
type MigrationOp =
  | { op: "add"; path: string[]; value: unknown }
  | { op: "remove"; path: string[] }
  | { op: "rename"; from: string[]; to: string[] }
  | { op: "setDefault"; path: string[]; value: unknown }
  | { op: "dataMigrate"; handler: (root: any) => void | Promise<void> }
```

The CLI [`zen db generate`](/api/cli/zen#zen-db) emits these for you; you only hand-write a `dataMigrate` op when the change requires custom logic.
