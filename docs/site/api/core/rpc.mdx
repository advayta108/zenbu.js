# `@zenbujs/core/rpc`

For most plugin code there is no `@zenbujs/core/rpc` import — you use [`useRpc()`](/api/core/react#userpc) on the renderer and direct service deps on the main process. This page documents the small advanced surface for users who need to interact with the RPC layer directly.

## Imports

```typescript
import type { ServiceRouter } from "@zenbujs/core/registry"
```

That's the entire normal-case import. The `ServiceRouter` type is the auto-derived shape of every registered service's RPC surface; you'll see it in selector signatures.

## What's *not* in `@zenbujs/core/rpc`

- `connectRpc` — used internally by `<ZenbuProvider>`. **Not public.**
- `EventProxy`, `RouterProxy` — internal client types.
- `EffectResult`, `ExtractRequirements`, `RpcContext`, `SerializedError` — Effect-flavored internals. **Not public.**
- `createServer`, `createClient`, `createRpcRouter` — the wire layer. **Not public.**

If you need to embed the RPC layer in a non-Zenbu renderer (e.g. a separate Vite app), open an issue describing your use case — we may add a `@zenbujs/core/rpc/transport` subpath.

## React-side hooks

These live in `@zenbujs/core/react`:

```typescript
import { useRpc, useEvents } from "@zenbujs/core/react"
```

### `useRpc()`

Returns a typed proxy over the registered services. Each property is keyed by service `static key`; each method is the service's async signature.

```typescript
function useRpc(): RouterProxy<ServiceRouter>
```

Calls return Promises. Errors thrown server-side reject the Promise with a plain `Error` whose message matches.

### `useEvents()`

Returns a typed event-subscribe proxy.

```typescript
function useEvents(): EventProxy<RegisteredEvents>
```

Use as: `events.<plugin-name>.<event-name>.subscribe(cb)`. The subscribe function returns an unsubscribe.

## Server-side surface

Inside a service:

- **Public async methods are auto-exposed.** No registration required.
- **`this.ctx.rpc`** (if you depend on `RpcService`) gives you the same emit proxy `Events` exposes — but you should use the [`Events`](/rpc/events) service directly. `RpcService` is intended as an internal coupling point.

If you want to call other services from your service, depend on them as a dep — not via the RPC router. See [Dependencies](/runtime/dependencies).

## Serialization

The framework's serializer supports:

| Type | Serialization |
| --- | --- |
| `string`, `number`, `boolean`, `null` | JSON-native |
| `Array`, plain object | JSON-native |
| `undefined` | dropped (treated as missing) |
| `Date` | ISO 8601 string + tag |
| `Uint8Array` | base64 string + tag |

Not supported: `bigint`, `Map`, `Set`, classes, functions. See [Calling procedures — what you can't do](/rpc/calling-procedures#what-you-can%27t-do).
