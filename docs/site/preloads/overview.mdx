# Preloads overview

Some data is hot enough that you want it ready **before any service starts** — and **before any renderer connects**. The framework calls this a **preload**.

Examples:

- The user's identity / session token, decoded from disk.
- Workspace paths and a `package.json` snapshot for the current project.
- A list of installed binaries, environment-variable lookups, OS information.
- A cached computation that takes 30ms but blocks every service if not ready.

A preload is a function the framework runs once during boot. Its return value is:

- **Frozen.** Preload outputs don't change at runtime. If you need mutable shared state, use [the database](/db/overview).
- **Available everywhere.** Services see it via `this.ctx.preloads.<plugin-name>`. Renderers see it via `usePreload()`.
- **Typed end-to-end.** The return type flows through the [registry](/rpc/typed-registry) automatically.

## When to use a preload

Preloads are for hot-path data with a few characteristics:

- It's **expensive enough** that paying for it in a service's `start()` is too slow (every dependent waits).
- It's **stable enough** that recomputing it would require an app restart anyway (env vars, OS info, identity).
- It's **read-only** to consumers. If anyone wants to *change* it, that's a database write or an event, not a preload.

If the data fails any of those, prefer the database (state) or RPC (request/response).

## When **not** to use a preload

- Anything that needs to react to changes — use database fields with `useDb`.
- Per-renderer state — use `useState`, or per-renderer DB sections.
- Heavy data that's rarely read — load it lazily on first call instead.
- Anything async-heavy that depends on user interaction — that's just an RPC.

## The shape of a preload

A plugin's preload is a single TypeScript file whose default export is a function:

```typescript
// src/main/preload.ts
import os from "node:os"

export default async function preload() {
  return {
    homeDir: os.homedir(),
    platform: process.platform,
    nodeVersion: process.version,
    bootedAt: Date.now(),
  }
}
```

Then point the manifest at it:

```json
{
  "name": "my-plugin",
  "preload": "src/main/preload.ts",
  "services": ["src/main/services/*.ts"]
}
```

That's the entire contract. The framework runs the function on every boot, awaits the result, freezes it, and exposes it everywhere as `preloads["my-plugin"]`.
