# Calling procedures

The most common caller of an RPC procedure is a React component in a renderer. The framework also supports calls from other plugins and from CLI scripts (via `zen exec`).

## From a React component

```tsx
import { useRpc } from "@zenbujs/core/react"

function AddNoteButton() {
  const rpc = useRpc()
  return (
    <button onClick={() => rpc.notes.add("new note")}>
      Add note
    </button>
  )
}
```

`useRpc()` returns a typed proxy. Properties resolve to `(args) => Promise<ReturnType>` — so this works:

```tsx
const result = await rpc.notes.add("hello")
result.id           // typed string
```

### Inside `useEffect`

If you call RPC from `useEffect`, return a no-op cleanup unless you actually need cancelability:

```tsx
useEffect(() => {
  rpc.notes.list().then(notes => setLocal(notes))
}, [rpc])
```

For cancelable calls, use an `AbortController`-friendly wrapper or just check a `cancelled` flag:

```tsx
useEffect(() => {
  let cancelled = false
  rpc.notes.list().then(notes => { if (!cancelled) setLocal(notes) })
  return () => { cancelled = true }
}, [rpc])
```

In practice, **prefer `useDb`** over fetching via RPC. Anything that lives in the database is already on the renderer's replica.

### Inside event handlers

```tsx
async function onClick() {
  try {
    await rpc.notes.add("hello")
  } catch (err) {
    setError(err instanceof Error ? err.message : String(err))
  }
}
```

A rejected RPC means the server threw. The optimistic update has already been rolled back; you just need to show the user.

## From another plugin's service

Services don't need RPC to talk to each other — they're in the same process. Declare a dep and call directly:

```typescript
export class MyService extends Service.with({ notes: NotesService }) {
  static key = "my-service"

  async doSomething() {
    const note = await this.ctx.notes.add("from my-service")
  }
}
```

This is **strictly preferred over RPC** for cross-service calls on the main process, because:

- It's fully type-checked without going through the registry.
- It's synchronous-ish (no wire serialization) — methods can return non-JSON values.
- It participates in the lifecycle: if `NotesService` isn't started, your service waits.

If you must use the RPC router from a service, the [`runtime` singleton](/api/core/runtime#runtime) lets you inspect the router, but this is for debug tooling only — **always prefer dep injection.**

## From a CLI script

`zen exec` runs a TypeScript file with `rpc` and `events` pre-opened against a running app:

```bash
zen exec -e 'console.log(await rpc.notes.list())'
zen exec ./my-script.ts
```

```typescript
// my-script.ts
import { rpc } from "@zenbujs/cli/exec"   // ambient — provided by zen exec

const notes = await rpc.notes.list()
console.log(notes)
```

The script connects over the same websocket the renderer uses. It exits cleanly when your code returns; long-running scripts can `await new Promise(...)` to stay alive.

## Error handling

```typescript
try {
  await rpc.notes.delete("missing-id")
} catch (err) {
  err instanceof Error    // true
  err.message             // server-side message
  // err.stack: undefined (not transferred over the wire by default)
}
```

For richer error info during development, run with `ZENBU_VERBOSE=1` — server-side errors get logged to the dev console with full stacks.

## What you can't do

- **Pass functions or classes as arguments.** The serializer doesn't support them.
- **Call RPC inside a `db.update(...)` callback.** Use the `runtime.get(SomeService)` pattern instead, or — better — declare it as a dep.
- **Subscribe to a server-pushed stream over RPC.** That's what [events](/rpc/events) are for. RPC is request/response only.
