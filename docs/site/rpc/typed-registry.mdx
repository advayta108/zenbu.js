# Typed registry

`useRpc()`, `useEvents()`, and `useDb()` are all generic, but **you never pass a type argument**. The types come from a per-app registry generated by [`zen link`](/api/cli/zen) and merged via TypeScript module augmentation.

If you've used [TanStack Router's `Register` interface](https://tanstack.com/router/v1/docs/framework/react/guide/file-based-routing#registration) — same pattern.

## What `zen link` does

`zen link` reads your app's `config.json` and every plugin's manifest, then writes:

```
<your-app>/types/
├── services.ts            # imports every plugin's services and exposes types
├── db-sections.ts         # imports every plugin's schema and merges them
├── events.ts              # imports every plugin's events.ts and merges them
├── preloads.ts            # imports every plugin's preload.ts and merges them
└── zenbu-register.ts      # declare-merges the above into ZenbuRegister
```

The last file is the magic:

```typescript
// types/zenbu-register.ts (generated)
declare module "@zenbujs/core/registry" {
  interface ZenbuRegister {
    db: DbRoot
    rpc: ServiceRouter
    events: PluginEvents
    preloads: Preloads
  }
}
```

This declare-merge tells TypeScript: when you see `useDb()`, the return type is `DbRoot`. When you see `useRpc()`, it's a proxy over `ServiceRouter`. And so on.

## When does it run

Three triggers:

- **Manually**: `nr zen link` — useful for CI.
- **On dev-server start**: the framework runs it before booting.
- **On manifest change**: the framework watches `zenbu.plugin.json` files and re-runs the generation.

You don't normally invoke it. If your IDE shows stale types after editing a manifest, restart TypeScript server.

## Registry hygiene

The generated files import every plugin's source. That means:

- A typo in any plugin's schema/services/events file is a TypeScript error in your app, even if the plugin isn't relevant to what you're working on.
- The registry uses **type-only imports**, so there's no runtime overhead — the registry isn't bundled into the renderer.

## What the augmentation point looks like

If you're writing a tool that wants to introspect the registry, the canonical entrypoint is:

```typescript
import type { ZenbuRegister } from "@zenbujs/core/registry"

type DbRoot = ZenbuRegister extends { db: infer T } ? T : never
type ServiceRouter = ZenbuRegister extends { rpc: infer T } ? T : never
type Events = ZenbuRegister extends { events: infer T } ? T : never
type Preloads = ZenbuRegister extends { preloads: infer T } ? T : never
```

These are the same conditional types `useDb`, `useRpc`, `useEvents`, and `usePreload` use internally.

## What if no `zen link` has run?

When `ZenbuRegister` has no augmentation, the hooks fall back to the framework's core types:

- `useDb()` returns `{ plugin: { core: CoreSchemaRoot } }` — the framework's own section only.
- `useRpc()` returns the framework's own RPC surface (which is empty by design — every method on built-in services starts with `_` to keep them off the wire).
- `useEvents()` returns `CoreEvents` — currently just `{ advice: { reload: { scope: string } } }`.

This is a reasonable fallback for "your plugins haven't generated yet" but you should rerun `zen link` before pushing.

## Why module augmentation, not generics

Two reasons:

1. **End-user ergonomics.** `useDb<MyRoot>(selector)` once is fine. Doing it in 60 components — and remembering to update it when you rename a plugin — is a maintenance tax.
2. **Cross-plugin types compose for free.** If your plugin imports another plugin's services and you've both run `zen link`, the resulting types are merged automatically. No generic juggling.

The trade-off is that *one* TypeScript project shares one `ZenbuRegister` augmentation. If you have two apps in the same monorepo with different plugin sets, each app's registry overrides the other's in TS — usually fine because tsconfig project boundaries keep them separate, but worth knowing.
