# Events

Events are how a service tells **every connected renderer** that something happened. They're for one-shot signals — `userLoggedIn`, `buildFinished`, `incomingMessage`. They are *not* a state model: the framework doesn't replay missed events, doesn't queue them on disconnect, and doesn't track subscribers per-renderer.

If the data has lifetime longer than "right now," put it in [the database](/db/overview). If a renderer needs to react to a one-time signal, emit an event.

## Declaring events

Each plugin owns its event topics. By convention they live at `src/main/events.ts`:

```typescript
// src/main/events.ts
export type Events = {
  noteAdded: { id: string; text: string; at: number }
  noteDeleted: { id: string }
}
```

Point the manifest at it:

```json
{
  "name": "notes-plugin",
  "events": "src/main/events.ts",
  // ...
}
```

That file is **purely a type declaration** — no runtime emit/subscribe API to wire up. The framework infers the plugin's namespace from `manifest.name` and produces:

- A typed `emit` proxy on the [`Events`](/api/core/runtime#built-in-services) service: `events["notes-plugin"].noteAdded(payload)`.
- A typed subscribe proxy in `useEvents()`: `events["notes-plugin"].noteAdded.subscribe(cb)`.

## Emitting

In any service that depends on the built-in `Events` service:

```typescript
import { Service, runtime } from "@zenbujs/core/runtime"
import { DbService, Events } from "@zenbujs/core/services"

export class NotesService extends Service.with({ db: DbService, events: Events }) {
  static key = "notes"

  async add(text: string) {
    const id = nanoid()
    await this.ctx.db.update(r => r.plugin["notes-plugin"].notes.concat([{ id, text }]))
    this.ctx.events["notes-plugin"].noteAdded({ id, text, at: Date.now() })
    return { id }
  }
}
```

`emit` is fan-out: every connected renderer gets the event. There is no targeted emit by default — if you want one-renderer-only, plumb a windowId through the payload and filter on the client.

<Note>
**Why is the namespace `["notes-plugin"]` and not `notes`?** Plugin event namespaces are derived from `manifest.name` to guarantee they don't collide across plugins. Two plugins can both export an `Events` type with a `tick` field; subscribers see them as `events["plugin-a"].tick` vs `events["plugin-b"].tick`.

If you want a shorter handle, alias it once at the top of your file: `const myEvents = events["my-plugin-name"]`.
</Note>

## Subscribing

In a React component:

```tsx
import { useEffect } from "react"
import { useEvents } from "@zenbujs/core/react"

function NotesList() {
  const events = useEvents()

  useEffect(
    () =>
      events["notes-plugin"].noteAdded.subscribe(({ id, text }) => {
        console.log("new note", id, text)
      }),
    [events],
  )

  // ...
}
```

`subscribe(cb)` returns an unsubscribe function. Returning it from `useEffect` cleans up correctly.

The callback receives the payload exactly as emitted — no wrapping, no metadata.

## Subscribing from outside React

If you have a non-React renderer (e.g. a background tab or an embedded view that doesn't render React), you can use `events` directly inside the `<ZenbuProvider>` boundary:

```tsx
// Custom hook for non-React subscribers, if you need it:
import { useEvents } from "@zenbujs/core/react"

function useNoteLogger() {
  const events = useEvents()
  useEffect(() => events["notes-plugin"].noteAdded.subscribe(logNote), [events])
}
```

`useEvents()` returns the same proxy in all components; subscribing twice is fine, the framework dedupes.

## Subscribing from another service

Services on the main process can also subscribe to events emitted by other services:

```typescript
import { Service, runtime } from "@zenbujs/core/runtime"
import { Events } from "@zenbujs/core/services"

export class AuditService extends Service.with({ events: Events }) {
  static key = "audit"

  async start() {
    this.useDisposable("audit-notes", () =>
      this.ctx.events["notes-plugin"].noteAdded.subscribe(({ id }) => {
        this.log("note added:", id)
      }),
    )
  }
}
```

Same proxy, same shape. The runtime delivers the event to all subscribers, regardless of which process they're in.

## Anti-patterns

- **Don't use events for state.** If a renderer reloads, it's missed every event up to that point. Anything important goes in the database.
- **Don't put large payloads in events.** Events go over the same websocket as RPC and DB sync — a 5MB blob in `noteAdded` blocks every other message until it's flushed. Reference a blob id instead.
- **Don't fire events as a side effect of `db.update`.** Emit *after* the update resolves; otherwise renderers might see the event before they see the update.
