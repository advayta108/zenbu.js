# Advice overview

Advice is the mechanism by which one plugin can **modify another plugin's renderer code at runtime** without forking it. Two flavors:

- **[Function advice](/advice/function-advice)** — wrap, replace, or hook a specific exported function in another plugin's source. Inspired by [Emacs's `advice-add`](https://www.gnu.org/software/emacs/manual/html_node/elisp/Advising-Functions.html) and aspect-oriented programming.
- **[Content scripts](/advice/content-scripts)** — inject a JavaScript module into another plugin's view. Inspired by browser extensions.

Both are scoped to a specific **view** — the view's `id` (e.g. `"notes"`) is what you target. Both can be registered and unregistered at runtime. Both reload their target views via the `advice.reload` event when registrations change.

## Mental model

```mermaid
flowchart LR
  pluginA[Plugin A<br/>registers a view 'notes']
  pluginB[Plugin B<br/>has no UI of its own]
  pluginC[Plugin C<br/>also extends 'notes']

  pluginB -. advise: replace App.tsx#Counter .-> view
  pluginC -. inject: src/content/clock.tsx .-> view

  view{{Notes view<br/>(running iframe)}}
```

Plugins B and C don't need to know about each other; both target the `notes` scope. Plugin A doesn't need to know B and C exist — its source isn't modified, only its *runtime behavior* is.

## When to use advice

- Your plugin needs to add a UI element to another plugin's view → [content script](/advice/content-scripts).
- Your plugin needs to override a function in another plugin's source → [function advice](/advice/function-advice).
- Your plugin needs to observe calls to a function (logging, instrumentation) → function advice with `kind: "before"` or `"after"`.

## When *not* to use advice

- You can use a normal RPC call → use RPC.
- You can subscribe to an event → use [events](/rpc/events).
- You're modifying your own plugin → just edit the source.
- You need to inject behavior into multiple unrelated plugins → write your code as a service that hooks into well-defined extension points instead.

Advice is powerful because it doesn't require the targeted plugin to opt in, but that power comes with a sharp edge: **you're targeting implementation, not an API**. If the targeted plugin renames a function or moves it to a different file, your advice silently stops applying. Use the higher-level mechanisms when they fit.

## Lifecycle

Both `advise(...)` and `inject(...)` return an unregister function. The recommended pattern wraps them in `useDisposable`:

```typescript
this.useDisposable("inject-clock", () =>
  inject({ view: "notes", module: "src/content/clock.tsx" }, import.meta),
)
```

When the registration is added or removed, the framework invalidates the targeted view's prelude module and emits an `advice.reload` event. The view receives the event, reloads, and the prelude script picks up the new advice/content-script set.
