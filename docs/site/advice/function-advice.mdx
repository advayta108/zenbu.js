# Function advice

Function advice modifies a single exported function in another plugin's source code. The framework's Babel transform rewrites every top-level function/component declaration so it dispatches through an advice chain — that chain is what `advise(...)` mutates.

## API at a glance

```typescript
import { advise } from "@zenbujs/core/advice"

const remove = advise({
  view: "notes",
  target: { module: "App.tsx", export: "Counter" },
  kind: "around",
  handler: { module: "src/content/wrap-counter.tsx", export: "WrapCounter" },
}, import.meta)
```

| Field | Meaning |
| --- | --- |
| `view` | The target view's scope. Use `"*"` for all views. |
| `target.module` | The module path within the targeted plugin (relative to its source root, no extension). |
| `target.export` | The exported function name. |
| `kind` | `"replace"`, `"before"`, `"after"`, or `"around"`. |
| `handler.module` | Your handler module path (relative to **your** plugin root, or absolute). |
| `handler.export` | The exported function in your handler module. |

The second arg, `import.meta`, lets the framework resolve `handler.module` against your plugin root.

## The four kinds

### `replace`

Drops the original function entirely; the handler is called with the same arguments.

```typescript
// src/content/wrap-counter.tsx
export function WrapCounter() {
  return <div>I'm not a counter anymore.</div>
}
```

### `before`

Runs the handler with the same arguments, then runs the original. The handler's return value is ignored.

```typescript
export function WrapCounter(props: any) {
  console.log("Counter called with", props)
}
```

### `after`

Runs the original first, then the handler with the original's return value.

```typescript
export function WrapCounter(props: any, originalResult: any) {
  console.log("Counter returned", originalResult)
}
```

### `around`

The most powerful kind. The handler receives a `next` function that calls the original (or skips it). Use it for caching, retry, conditional behavior, full wrapping.

```typescript
import type { ReactElement } from "react"

export function WrapCounter(props: any, next: (...args: any[]) => ReactElement) {
  const original = next(props)
  return <div className="wrapped">{original}</div>
}
```

`next(...args)` is *the function the chain would call without your advice*. If multiple `around` advices are stacked, calling `next()` invokes the next-inner one; the innermost `next()` calls the original.

## Targeting

`target.module` is the path of the file in the targeted plugin where the export lives. For example, if the targeted view's source is in `plugins/notes-plugin/src/renderer/App.tsx` and you want to advise the `Counter` export:

```typescript
target: { module: "App.tsx", export: "Counter" }
```

The framework resolves `App.tsx` relative to the targeted view's renderer root — same paths the view's `import` statements use.

`target.export` must be the literal exported name. Default exports are referenced as `default`:

```typescript
target: { module: "Header.tsx", export: "default" }
```

## What functions can be advised

The advice transform rewrites:

- Top-level `function` declarations.
- Top-level `const x = (...) => ...` arrow assignments.
- Top-level `export const x = ...` arrow assignments.
- React components by virtue of being one of the above.

It does **not** rewrite:

- Class methods.
- Functions defined inside another function.
- Object-literal methods (`{ foo() {} }`).
- Symbols re-exported from other modules.

If you target a function the transform didn't rewrite, the registration silently no-ops. Use `nr zen advice list <view>` to see which functions are advisable.

## Stacking and order

Multiple advices on the same target compose:

- All `before` handlers run first, in registration order.
- Then `around` handlers run, with the innermost being the original.
- Then `after` handlers run, in registration order.
- A `replace` short-circuits everything else.

For multiple `replace` advices on the same target, the most-recently-registered wins. This lets a higher-priority plugin override a lower-priority one.

## Lifecycle

`advise(...)` returns an unregister function. Wrap it in `useDisposable`:

```typescript
this.useDisposable("counter-advice", () =>
  advise({...}, import.meta),
)
```

When registrations change, the targeted view receives an `advice.reload` event and reloads (full page).

## Common patterns

### Add a debug overlay around every component

```typescript
this.useDisposable("debug", () =>
  advise({
    view: "*",
    target: { module: "*", export: "*" },     // not yet supported — see below
    kind: "around",
    handler: { module: "src/content/debug-wrap.tsx", export: "DebugWrap" },
  }, import.meta),
)
```

<Note>
Wildcard module/export targeting is on the roadmap; today you have to register one advice per function.
</Note>

### Replace a button with a different one

```typescript
this.useDisposable("better-add-note", () =>
  advise({
    view: "notes",
    target: { module: "AddNoteButton.tsx", export: "AddNoteButton" },
    kind: "replace",
    handler: { module: "src/content/better-button.tsx", export: "BetterButton" },
  }, import.meta),
)
```

### Log every call to a service method (renderer-side)

This needs to live where the call happens — usually in a custom hook or component. Adviseable function targets are renderer-side; if you want to log RPC calls, the right tool is a wrapper around `useRpc()` (or, on the main process, a wrapping service).

## Limitations

- Advice **only applies in renderers**, not on the main process. Main-process service methods can be wrapped by writing a wrapper service that depends on the original.
- Advice **only matches by module path + export name**, not by symbol identity. If the targeted plugin renames the export, your advice silently stops applying.
- Advice **cannot read the targeted plugin's local state**. You can use Zenbu hooks (`useDb`, `useRpc`) inside the handler, but you can't reach into the targeted plugin's React state.
