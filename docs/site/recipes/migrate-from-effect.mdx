# Migrate from Effect

The framework used to expose Effect-flavored APIs (`db.effectClient`, `KyjuError`, etc.) on its public surface. As of the first stable release, **the public API is plain Promises** â€” see [API design principles](/reference/api-design-principles) for why.

If your existing code used Effect, here's how to migrate.

## Database client

### Before

```typescript
import * as Effect from "effect/Effect"
import type { KyjuError } from "@zenbu/kyju"

await Effect.runPromise(
  this.ctx.db.effectClient.update(root => {
    root.plugin["my-plugin"].count += 1
  }),
)
```

### After

```typescript
await this.ctx.db.update(root => {
  root.plugin["my-plugin"].count += 1
})
```

`db.update`, `db.read`, `db.blobs.*`, `db.collection(...)` are all plain Promise-returning methods. No `Effect.runPromise` wrapping.

## Database errors

### Before

```typescript
const exit = await Effect.runPromiseExit(this.ctx.db.effectClient.update(...))
if (Exit.isFailure(exit)) {
  // exit.cause may contain KyjuError tagged unions
}
```

### After

```typescript
try {
  await this.ctx.db.update(...)
} catch (err) {
  err instanceof Error    // true
  err.message             // human-readable
}
```

Tagged-union errors (`InvalidSessionError`, `VersionMismatchError`, etc.) are gone from the public surface. The framework throws plain `Error`s with descriptive messages and the original cause attached as `err.cause`.

If you need to discriminate between failure modes, expose the case in your service code (a different RPC method, a different return type, or a sentinel value).

## RPC

### Before

```typescript
type RouterMethod = (...args: any[]) => Effect.Effect<Result, MyError>
```

The framework auto-converted Effect-returning methods over the wire. The renderer saw `Promise<EffectResult<A, E>>`.

### After

Just write `async` methods. They're auto-exposed as Promise-returning RPC. No Effect wrapping at the wire.

If you have a service method that genuinely uses Effect internally:

```typescript
async myMethod(arg: string): Promise<MyResult> {
  return Effect.runPromise(internalEffectComputation(arg))
}
```

Wrap once at the boundary; the public surface stays Promise-shaped.

## Why the change

Three reasons:

1. **Onboarding cost.** Plugin authors shouldn't have to learn Effect to ship a notes app.
2. **Consistency.** Half-Effect / half-Promise public APIs cause drift over time as new symbols land in one style or the other.
3. **The framework's internals can still use Effect.** Removing it from the public surface doesn't change how kyju is implemented.

## Effect escape hatch (planned)

For users who actively want Effect-shaped APIs, we plan to add `@zenbujs/core/db/effect` as an opt-in subpath:

```typescript
import { effectDb } from "@zenbujs/core/db/effect"

const result = yield* effectDb.update(...)
```

Not yet shipped. If you'd benefit from this, file an issue describing the use case.

## Cleanup checklist

- [ ] Replace every `db.effectClient.<...>` with `db.<...>`.
- [ ] Drop every `Effect.runPromise(...)` wrapping a kyju call.
- [ ] Drop imports of `KyjuError`, `InvalidSessionError`, `VersionMismatchError`, `NotFoundError`, `ReferenceExistsError`.
- [ ] Update RPC methods that returned `Effect.Effect<...>` to return `Promise<...>`.
- [ ] Run `nr typecheck` and `nr lint` â€” anything that referenced the removed symbols will fail loudly.
