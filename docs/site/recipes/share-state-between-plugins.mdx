# Share state between plugins

The cleanest way for plugin A to read plugin B's state is through the database. Each plugin owns a section under `root.plugin.<name>`, and any plugin can read any section as long as it's installed.

## 1. Declare a peer-dep

In your `package.json`:

```json
{
  "peerDependencies": {
    "@example/notes-plugin": "*"
  }
}
```

This is npm-level — it makes the type imports work at compile time. The runtime doesn't actually require the npm package to be installed, only that the *plugin* is installed in the user's `config.json`.

## 2. Read from the database

```typescript
import { Service, runtime } from "@zenbujs/core/runtime"
import { DbService } from "@zenbujs/core/services"

export class NoteCounterService extends Service.with({ db: DbService }) {
  static key = "note-counter"

  async getCount(): Promise<number> {
    const root = this.ctx.db.read()
    return root.plugin["notes-plugin"]?.notes?.length ?? 0
  }
}
```

The optional chaining handles "the user uninstalled the notes plugin" gracefully.

## 3. Subscribe from the renderer

```tsx
import { useDb } from "@zenbujs/core/react"

function NoteCount() {
  const count = useDb(root => root.plugin["notes-plugin"]?.notes?.length ?? 0)
  return <span>{count} notes</span>
}
```

Because the replica fan-outs all sections to all renderers, this is just a `useDb` selector — no extra RPC, no extra subscriptions.

## What if I need to *write* to another plugin's section?

Don't write directly. Call the owning plugin's RPC:

```typescript
const rpc = useRpc()
await rpc["notes-plugin"].add("hello")
```

Cross-plugin writes via `db.update` work, but they bypass the owning plugin's validation, events, and any future migration logic. RPC keeps the contract honest.

## What if the other plugin has Effect / Promise variants?

There's only one public surface; no Effect variant exists. See [Migration recipe](/recipes/migrate-from-effect) if you have legacy code that used `db.effectClient`.
