# Add a schema field

Adding a field to a Kyju schema is two steps: edit the schema, generate a migration.

## 1. Edit the schema

```typescript
// src/main/schema.ts
import { defineSchema, f, z } from "@zenbujs/core/db"

export const schema = defineSchema({
  count: f.number().default(0),
  // New field:
  user: f.object({ name: z.string(), email: z.string().optional() }).default({ name: "" }),
})
```

The `.default(...)` is what the migration generator inserts for existing rows.

## 2. Generate the migration

```bash
nr zen db generate --name add_user
```

The generator writes `db/migrations/<plugin-name>/0002_add_user.ts`:

```typescript
import type { KyjuMigration } from "@zenbujs/core/db"

export const migration: KyjuMigration = {
  name: "add_user",
  ops: [
    { op: "add", path: ["user"], value: { name: "" } },
  ],
}

export default migration
```

Commit the file. It's part of your source.

## 3. Save and reload

The dev server's `DbService` re-evaluates with the new schema, sees the pending migration, applies it, and broadcasts the new root to renderers. Your `useDb` selectors that touch `user` start working immediately.

## Common variations

### Add a nullable field

```typescript
user: f.object({...}).nullable().default(null)
```

The migration adds `user: null`.

### Add a collection

```typescript
notes: f.collection(z.object({ id: z.string(), text: z.string() })),
```

The migration creates the collection's storage. Existing items are unaffected — collections are addressed by id, so no data migration is needed.

### Add a blob

```typescript
thumbnail: f.blob({ debugName: "thumbnail" }),
```

The migration adds the field. The blob bytes are missing until you write a value.

### Renaming a field

The generator detects renames if the new field's name and shape are similar enough:

```typescript
// Before
old: f.string().default("")

// After
new: f.string().default("")
```

→ generator emits `{ op: "rename", from: ["old"], to: ["new"] }`. Verify the generated file looks right before committing.

### Schema-incompatible change

If you're changing a field's type (`string` → `number`), the generator emits a warning and writes a `dataMigrate` placeholder you have to complete:

```typescript
ops: [
  { op: "remove", path: ["count"] },
  { op: "add", path: ["count"], value: 0 },
  {
    op: "dataMigrate",
    handler: (root) => {
      // TODO: convert old string to new number
    },
  },
]
```
