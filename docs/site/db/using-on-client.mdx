# Using on the client

On the renderer, the database is exposed through three React hooks. They all live in `@zenbujs/core/react`.

Every hook must be called inside a `<ZenbuProvider>` — the framework wraps the renderer entrypoint with it automatically, so you almost never set this up yourself. (See [Customizing boot](/boot/customizing-boot) if you do need to.)

## `useDb(selector)`

Subscribe to a slice of the root. The selector receives the root replica; whatever it returns is what your component re-renders on changes to.

```tsx
import { useDb } from "@zenbujs/core/react"

function Counter() {
  const count = useDb(root => root.plugin["counter-app"].count)
  return <span>{count}</span>
}
```

Re-renders are scoped: this component re-renders only when `root.plugin["counter-app"].count` changes, not when other parts of the root update. Comparison is shallow on the selector return value (or you can pass a custom equality function as the second arg).

If you need the entire root:

```tsx
const root = useDb()
```

(Useful for debugging panels; not common in normal app code.)

## `useCollection(ref)`

Subscribe to a kyju collection. Pass a `CollectionRefValue` — typically obtained via `useDb`:

```tsx
import { useDb, useCollection } from "@zenbujs/core/react"

function NoteList() {
  const ref = useDb(root => root.plugin["notes-plugin"].notes)
  const { items, concat } = useCollection(ref)

  return (
    <div>
      {items.map(n => <div key={n.id}>{n.text}</div>)}
      <button onClick={() => concat([{ id: nanoid(), text: "new", createdAt: Date.now() }])}>
        Add
      </button>
    </div>
  )
}
```

The hook returns:

- `items: Item[]` — currently-loaded items, in collection order.
- `collection: CollectionNode | null` — the underlying client collection (advanced).
- `concat(items: Item[]): void` — optimistic append.

See [Collections](/db/collections) for the storage model.

## `useBlob(blobId)`

Fetch a blob's bytes:

```tsx
import { useDb, useBlob } from "@zenbujs/core/react"

function Thumbnail() {
  const ref = useDb(root => root.plugin["my-plugin"].thumbnail)
  const data = useBlob(ref?.blobId)

  if (!data) return <span>Loading…</span>

  const url = URL.createObjectURL(new Blob([data]))
  return <img src={url} />
}
```

Returns `Uint8Array | null`. While loading, you get `null`. If `blobId` is undefined or the blob doesn't exist, you get `null` permanently.

You're responsible for revoking the object URL when the component unmounts. (We may add a `useBlobUrl` helper that handles this for you.)

## Lifecycle and connection states

`<ZenbuProvider>` manages the websocket and the replica. The renderer experiences three connection states:

| State | What's true | UX behavior |
| --- | --- | --- |
| `connecting` | Initial connect or reconnect in progress | Renders the `fallback` prop or a transparent placeholder |
| `connected` | Wire is up; replica is hydrated | Children render; hooks work |
| `error` | Reconnect attempts exhausted | Renders the `errorFallback` prop or nothing |

In dev, you'll occasionally flicker through `connecting` when the main-process services reload — the websocket drops, the new RPC router builds, and the renderer reconnects. This is by design, not a bug.

## Optimistic writes (and how rollback feels)

When you call `rpc.<service>.method(...)` from a renderer, and that method does `db.update(...)` inside, the **renderer's replica applies the same update locally before the wire resolves**. So:

```tsx
function MyButton() {
  const rpc = useRpc()
  return <button onClick={() => rpc["counter-app"].increment()}>+1</button>
}
```

— the count visually increments before the server has confirmed.

If the server's `db.update` throws (e.g. a validation check failed), the renderer rolls back. You'll typically see this as the UI snapping back to the prior state with no animation. If you want to handle it, await the RPC and act on the rejection:

```tsx
try {
  await rpc["counter-app"].increment()
} catch (err) {
  alert(err.message)
}
```

In practice, validation rejections are rare in a single-user app, because there's no auth to disagree about. If you find yourself relying on rollback for UX, that's usually a sign you should validate optimistically on the client too.

## Performance notes

- **Selector return value should be stable.** A new object each render means a re-render every push, even if the data didn't change. Either select primitives or memoize.
- **Avoid arrays of arrays in `useDb` selectors.** The shallow-equality fast path doesn't bottom out into nested arrays. Select at one level deeper.
- **Don't put 100k items in `useDb`.** That's what `useCollection` is for. The replica's `useDb` reads from the same in-memory copy as `update` writes to, so big sections cost both memory and re-render time.
