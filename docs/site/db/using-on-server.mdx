# Using on the server

In any service that depends on [`DbService`](/api/core/runtime#built-in-services), you have a typed client at `this.ctx.db`.

```typescript
import { Service, runtime } from "@zenbujs/core/runtime"
import { DbService } from "@zenbujs/core/services"

export class MyService extends Service.with({ db: DbService }) {
  static key = "my-service"

  // ...
}
```

## Reading

`db.read()` returns a synchronous snapshot of the entire root:

```typescript
const root = this.ctx.db.read()
const count = root.plugin["my-plugin"].count
```

The returned object is a plain frozen object — mutations do nothing. Always go through `db.update(...)` for writes.

The snapshot is **stable**: if you `read()` and then await for ten seconds, the object you held still reflects the state at read time, not the current state. If you need the latest, call `read()` again.

## Writing

`db.update(fn)` runs your function against a draft of the root and applies the diff:

```typescript
await this.ctx.db.update(root => {
  root.plugin["my-plugin"].count += 1
})
```

Rules:

- **The function may be async.** The transaction commits on resolution.
- **If the function throws, nothing is committed.** Error bubbles up.
- **Mutate in place.** Don't return a new object — Kyju observes mutations on the draft.
- **Multiple writes are batched.** The diff is computed once, broadcast once, and applied to replicas as a single update.
- **Concurrent updates are serialized.** Two `db.update` calls run in order; you'll never see a half-applied root.

## Cross-section reads

Your section is `root.plugin["<your-plugin-name>"]`. You can read other plugins' sections too, as long as a peer-dep is declared:

```typescript
const settings = root.plugin["settings-plugin"].theme
```

The type is the union of every registered plugin's section, populated by [`zen link`](/api/cli/zen). If a plugin isn't registered at type-check time, that key is missing from the type — runtime access still works but TypeScript won't help you.

## Cross-section writes

Don't write to another plugin's section directly, even if you can:

```typescript
// 🚫 don't do this:
await this.ctx.db.update(root => {
  root.plugin["settings-plugin"].theme = "dark"
})
```

Instead, RPC into the owning plugin:

```typescript
const settings = runtime.get(SettingsService)        // or declare as a dep
await settings.setTheme("dark")
```

Why: the owning plugin is responsible for invariants on its own data. Direct cross-plugin writes circumvent its validation, change events, and any future migration logic.

## Blobs and collections

```typescript
// Blobs:
const blobId = await this.ctx.db.blobs.create(data)
const data2 = await this.ctx.db.blobs.read(blobId)
await this.ctx.db.blobs.delete(blobId)

// Collections:
const ref = root.plugin["my-plugin"].notes
const reader = this.ctx.db.collection(ref)
const items = await reader.all()
await reader.delete("note_3")
```

See [Blobs](/db/blobs) and [Collections](/db/collections).

## Common patterns

### Read-modify-write with validation

```typescript
async incrementIfBelow(max: number) {
  await this.ctx.db.update(root => {
    const cur = root.plugin["my-plugin"].count
    if (cur >= max) {
      throw new Error(`Counter already at ${cur}`)
    }
    root.plugin["my-plugin"].count = cur + 1
  })
}
```

If the check fails, the throw bails out the whole transaction — no partial write.

### Append to a collection from RPC

```typescript
async addNote(text: string) {
  const id = nanoid()
  await this.ctx.db.update(root => {
    root.plugin["notes-plugin"].notes.concat([
      { id, text, createdAt: Date.now() },
    ])
  })
  return { id }
}
```

The renderer's optimistic write applies locally before the RPC resolves. The server's broadcast lands a moment later and confirms.

### Fanout to events

```typescript
async incremented() {
  await this.ctx.db.update(root => { root.plugin["my-plugin"].count += 1 })
  this.ctx.events["my-plugin"].incremented({ at: Date.now() })
}
```

State changes go through the database; one-shot signals go through events. Don't conflate them. See [Events](/rpc/events).

## What to avoid

- **Reading via async wrappers.** `db.read()` is sync. There is no `db.readAsync()`.
- **Holding a snapshot across awaits and assuming it's fresh.** Re-read after every await.
- **Mutating outside `db.update(...)`.** Mutations on the snapshot are silently dropped.
- **Calling `db.update` from inside another `db.update` callback.** It deadlocks. Compose your changes into a single update.
