# Blobs

A **blob** field holds a `Uint8Array` of arbitrary size. The bytes live in their own file under `<dbPath>/blobs/`, addressed by a content hash; the schema field stores only the blob's id.

Use blobs for:

- thumbnails, images, audio,
- files the user dropped onto the app,
- any payload that's too big to keep in the section JSON (which is loaded into memory in every replica).

## Declaring a blob field

```typescript
import { defineSchema, f } from "@zenbujs/core/db"

export const schema = defineSchema({
  thumbnail: f.blob({ debugName: "thumbnail" }),
})
```

`debugName` is what shows up in tooling output ([`zen kyju db root`](/api/cli/zen)) and inspector views. It doesn't affect runtime behavior.

## Writing a blob

```typescript
const data = new TextEncoder().encode("hello world")
const blobId = await this.ctx.db.blobs.create(data)

await this.ctx.db.update(root => {
  root.plugin["my-plugin"].thumbnail = { blobId, debugName: "thumbnail" }
})
```

`db.blobs.create(data)` writes the bytes and returns a stable id (a content hash). The blob exists on disk **before** you reference it from the section, so it's safe across crashes — orphan blobs are GC'd by a background sweep.

## Reading a blob

```typescript
const ref = root.plugin["my-plugin"].thumbnail
const data = await this.ctx.db.blobs.read(ref.blobId)
// data is Uint8Array | null
```

On the renderer:

```typescript
import { useBlob } from "@zenbujs/core/react"

const ref = useDb(root => root.plugin["my-plugin"].thumbnail)
const data = useBlob(ref?.blobId)              // Uint8Array | null
```

`useBlob` handles the round-trip: it issues a chunked fetch over the websocket, returns `null` while loading, and re-fetches on blob-id change.

## Deleting a blob

```typescript
await this.ctx.db.blobs.delete(blobId)
```

This is rarely the right call. Blobs are content-addressed: deleting one means breaking every reference to it. For most apps you should:

1. Replace the section field's reference (`root.plugin[...].thumbnail = newRef`).
2. Let the GC sweep delete the orphan when no live reference points at it.

Manual `delete()` is for cases where you need to free disk *now*.

## Hot blobs

By default, blob writes go to disk synchronously. Pass `hot: true` if you want a write that's only kept in memory (useful for short-lived previews):

```typescript
const blobId = await this.ctx.db.blobs.create(data, { hot: true })
```

Hot blobs survive process lifetime but disappear on shutdown. They're not GC'd while in use.

## Where blobs live on disk

```
<dbPath>/
└── blobs/
    └── ab/cd/ab_cdef0123…   # content-hash sharded
```

The sharding (first byte / second byte) keeps any one directory under ~64K entries. Every blob is immutable; "updates" produce a new id.
