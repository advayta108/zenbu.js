# Quickstart

By the end of this page you'll have a Zenbu desktop app with a counter that:

- persists across restarts,
- updates every open window optimistically and instantly,
- recompiles its main process and its UI on file save without restarting.

We'll write **all of it in TypeScript**; no IPC handlers, no Redux, no SQL.

## Prerequisites

You need:

- **Node 22+** (for the CLI)
- **A working Electron build**. If you've never run an Electron app, `npx electron --version` should print a version. The first time you run a Zenbu app the framework will install Electron + bundled `pnpm` + `bun` into `~/Library/Caches/Zenbu` (macOS) or the equivalent on Linux/Windows.

## 1. Scaffold the app

```bash
npx create-zenbu-app counter-app
cd counter-app
ni
```

This drops a working app on disk. The structure looks like:

```
counter-app/
├── package.json
├── tsconfig.json
├── zenbu.plugin.json     # the manifest — your app is a plugin
├── config.json           # which plugins this app loads (just yours, for now)
├── vite.config.ts
├── electron-builder.json
├── zenbu.build.ts        # build pipeline (see /build)
└── src/
    ├── main/             # services
    │   └── services/
    │       └── app.ts    # one empty service to get started
    └── renderer/         # UI — a normal React + Vite app
        ├── App.tsx
        ├── main.tsx
        ├── index.html
        └── app.css
```

Run `nr dev` and a window should appear. It'll just say "Welcome to Zenbu" — we're going to make it more interesting.

<Tip>
**Tip:** Leave `nr dev` running. Every step below is something you can save and the change appears live, no restart.
</Tip>

## 2. Define the schema

Zenbu apps store data in a built-in document database called Kyju. We'll declare one field — a counter — and the framework takes care of migrations, persistence, and renderer sync.

Create `src/main/schema.ts`:

```typescript
import { defineSchema, f } from "@zenbujs/core/db"

export const schema = defineSchema({
  count: f.number().default(0),
})
```

That's it. `count` is now a typed, auto-migrated, default-initialized field. The default value flows into the migration generator and into the runtime's first-boot insert.

Now point the manifest at it. Open `zenbu.plugin.json` and add a `schema` field:

```json
{
  "name": "counter-app",
  "uiEntrypoint": "src/renderer",
  "services": ["src/main/services/*.ts"],
  "schema": "src/main/schema.ts"
}
```

Generate the migration the framework will use to introduce this field:

```bash
nr zen db generate --name init
```

A `db/migrations/0001_init.ts` file appears with the change description. Commit it — migrations are part of your source.

<Note>
The generated registry types update automatically as you save files; no extra `zen link` needed for this tutorial. If you ever need to force-regenerate them, run `nr zen link`.
</Note>

## 3. Write the service

Replace the contents of `src/main/services/app.ts`:

```typescript
import { Service, runtime } from "@zenbujs/core/runtime"
import { DbService } from "@zenbujs/core/services"

export class CounterService extends Service.with({ db: DbService }) {
  static key = "counter-app"

  async increment() {
    await this.ctx.db.update(root => {
      root.plugin["counter-app"].count += 1
    })
  }

  async reset() {
    await this.ctx.db.update(root => {
      root.plugin["counter-app"].count = 0
    })
  }
}

runtime.register(CounterService, import.meta)
```

Three things happened:

1. We declared a dependency on `DbService`. The framework guarantees `db` is started before `start()` ever runs.
2. We defined two `async` methods. They are **automatically** RPC-callable from the renderer as `rpc["counter-app"].increment()` and `rpc["counter-app"].reset()`.
3. We called `runtime.register(CounterService, import.meta)`. The `import.meta` argument is what makes this service hot-reloadable — when you save this file, the framework tears down the old instance and spins up the new one without restarting the app.

## 4. Render the UI

Replace `src/renderer/App.tsx`:

```tsx
import { useDb, useRpc } from "@zenbujs/core/react"

export function App() {
  const count = useDb(root => root.plugin["counter-app"].count)
  const rpc = useRpc()

  return (
    <main style={{ padding: 32, textAlign: "center", fontFamily: "system-ui" }}>
      <h1 style={{ fontSize: 64, margin: 0 }}>{count}</h1>
      <div style={{ display: "flex", gap: 12, justifyContent: "center", marginTop: 16 }}>
        <button onClick={() => rpc["counter-app"].increment()}>+1</button>
        <button onClick={() => rpc["counter-app"].reset()}>Reset</button>
      </div>
    </main>
  )
}
```

That's a complete Zenbu UI:

- `useDb(selector)` subscribes to a slice of the live database. The selector is typed against your schema (no `any`).
- `useRpc()` returns a typed proxy over your services. `rpc["counter-app"].increment()` returns a `Promise<void>` and is type-checked against the actual method signature on the server.
- No state management, no `useEffect`, no fetch hooks. The replica handles cache invalidation by being structurally identical to the source of truth.

Save and watch the window. The counter persists across restarts and across open windows.

## 5. Add an event

So far reads come through the database. Sometimes you want to broadcast a one-shot event ("user logged in", "build finished") instead of mutating state. Zenbu calls these **events**.

Create `src/main/events.ts`:

```typescript
export type Events = {
  incremented: { at: number; newValue: number }
}
```

Point the manifest at it:

```json
{
  "name": "counter-app",
  "uiEntrypoint": "src/renderer",
  "services": ["src/main/services/*.ts"],
  "schema": "src/main/schema.ts",
  "events": "src/main/events.ts"
}
```

Update the service to emit the event:

```typescript
import { Service, runtime } from "@zenbujs/core/runtime"
import { DbService, Events } from "@zenbujs/core/services"

export class CounterService extends Service.with({ db: DbService, events: Events }) {
  static key = "counter-app"

  async increment() {
    await this.ctx.db.update(root => {
      root.plugin["counter-app"].count += 1
    })
    const newValue = this.ctx.db.read().plugin["counter-app"].count
    this.ctx.events["counter-app"].incremented({ at: Date.now(), newValue })
  }
}

runtime.register(CounterService, import.meta)
```

And subscribe in the renderer:

```tsx
import { useEffect } from "react"
import { useEvents } from "@zenbujs/core/react"

export function App() {
  const events = useEvents()
  useEffect(
    () =>
      events["counter-app"].incremented.subscribe(({ newValue }) => {
        console.log("incremented to", newValue)
      }),
    [events],
  )
  // ...
}
```

`events` is fully typed; you can't subscribe to a topic you haven't declared.

## 6. (Optional) Build and run a binary

When you're ready to ship, the build pipeline produces a desktop binary:

```bash
nr build:source       # transform & stage your TS source
nr build:desktop      # bundle into a .app / .exe / .AppImage
```

By default the binary uses the [live-source adapter](/build/live-source-adapter) — it ships your TypeScript source, bundles `pnpm` + `bun`, and pulls future updates over `git`. To produce a traditional Electron app with no toolchain or update infrastructure, comment out one line in `zenbu.build.ts`:

```typescript
import { defineBuildConfig } from "@zenbujs/cli/build"
// import { liveSource } from "@zenbujs/cli/adapters/live-source"

export default defineBuildConfig({
  // adapter: liveSource({ mirror: { target: "you/counter-app-release" } }),
})
```

See [Build & distribution](/build/overview) for the full story.

## What you have now

In ~30 lines of TypeScript, you've built:

- a typed, persisted, auto-migrated database section,
- a hot-reloadable service that exposes typed RPC procedures,
- a typed event topic,
- a React UI with optimistic updates and live database subscriptions,
- a working production build pipeline (commented out for now).
